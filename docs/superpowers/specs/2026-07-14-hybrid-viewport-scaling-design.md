# 混合式视口缩放设计说明

## 目标

在浏览器缩放 100% 的桌面 Chromium 中，以 2560×1440 为视觉基准：字体、图标、间距、控制尺寸按同一个系数缩放；外壳和内容工作区使用原生 Grid/Flex 铺满窗口。页面不留未分配的底部空白，不裁切，不使用整页 `zoom` 或 `transform: scale()`。

## 已确认决策

- 视觉基准：2560×1440，缩放系数为 1。
- 1920×1080：重点验收尺寸，缩放系数为 0.75。
- 1366×768：继续按公式缩放，不设置最小缩放档。
- 3840×2160：缩放系数为 1.5，不设置最大缩放档。
- 非 16:9 窗口：取宽高比例的较小值进行视觉缩放，多出来的另一轴空间交给主工作区。
- 1px 边框、表格分隔线和焦点轮廓保持设备 CSS 像素，不参与比例缩放。
- 第一阶段只改配置中心样板页；参数经四档截图确认后才允许推广全站。

## 缩放模型

```text
uiScale = min(viewportWidth / 2560, viewportHeight / 1440)
```

| 视口 | uiScale | 剩余空间处理 |
|---|---:|---|
| 1366×768 | 0.533333 | 16:9，无剩余轴 |
| 1440×900 | 0.562500 | 多出的纵向空间进入工作区 |
| 1920×1080 | 0.750000 | 16:9，无剩余轴 |
| 2048×1024 | 0.711111 | 多出的横向空间进入主内容列 |
| 2560×1440 | 1.000000 | 设计基准 |
| 3840×2160 | 1.500000 | 16:9，无剩余轴 |

缩放只生成 CSS 设计令牌的最终像素值。DOM 不整体缩放，因此点击坐标、固定定位、滚动容器和可访问性树保持正常。

## 令牌模型

初始令牌以“1920 下尽量保持现有视觉密度”为校准锚点，再换算成 2560 基准值：

| 令牌 | 2560 基准 | 1920 结果 | 用途 |
|---|---:|---:|---|
| `--ui-font-xs` | 16px | 12px | 辅助信息 |
| `--ui-font-sm` | 17.3333px | 13px | 表格与导航 |
| `--ui-font-md` | 18.6667px | 14px | 正文与控件 |
| `--ui-font-lg` | 21.3333px | 16px | 小标题 |
| `--ui-font-xl` | 26.6667px | 20px | 页面标题 |
| `--ui-sidebar-width` | 352px | 264px | 主侧栏 |
| `--ui-topbar-height` | 77.3333px | 58px | 顶栏 |
| `--ui-control-sm` | 40px | 30px | 紧凑控件 |
| `--ui-control-md` | 48px | 36px | 标准控件 |
| `--ui-page-gutter` | 18.6667px | 14px | 页面边距 |

这些数值是样板页初始值，不是未经观察就冻结的最终值。四档截图中如 1366 文字不可读或 4K 密度过低，只调整 2560 基准令牌；缩放公式和无上下限规则保持不变。

## 布局模型

高度传递链固定为：

```text
100dvh app shell
  -> flex main
    -> flex route-content
      -> opt-in route-fill-page
        -> minmax(0, 1fr) workspace
          -> internal scroll region
```

- 工作区型页面使用 `.route-fill-page`，把额外高度交给表格主体、编辑区、图表或详情栏。
- 表单、上传、向导和预检页面保持内容自然高度，由路由容器负责页面滚动。
- 非 16:9 产生的额外宽度只增加主内容列；侧栏和操作栏继续使用缩放令牌。

## 推广策略

1. 配置中心三种模式完成缩放和铺满。
2. 四档截图与 DOM 指标通过后冻结第一版令牌。
3. 迁移共享外壳。
4. 按“中心/列表、工作台/结果、编辑器、自然高度流程”四类逐批迁移。
5. 每批只运行对应测试和浏览器矩阵，不执行无关全量测试。

## 停线条件

- 配置中心尚未通过四档视觉确认，不开始全站推广。
- 任一页面需要整页 `zoom`、`transform: scale()` 或裁切才能通过，停止并重新设计该页面。
- 任一视口出现横向页面级滚动、固定操作被遮挡、双重纵向滚动或文字明显不可读，停止该批迁移。
- 当前工作树包含大量既有未提交改动；全站阶段开始前必须先确定这些改动的归属，不能覆盖或混入无关修改。
