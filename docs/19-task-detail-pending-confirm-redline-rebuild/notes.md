# 调研、决策与迭代笔记

## 已确认事实

- 参考图实际尺寸为 1589×997。
- 本轮目标状态是“九类能力生成完成、等待人工确认”，不是旧任务书的空资料态或生成中态。
- 当前 `/tasks/detail` 由 `src/views/tasks/TaskDetail.vue` 实现，仍是通用台账布局，结构上缺少目标图的 3×3 结果卡、集中待确认表、底部版本/导出预览和完整右侧追溯抽屉。
- 技术栈是 Vue 3 + Vue Router + Vite，已有 Font Awesome 图标依赖。
- 仓库没有初始提交，且现有项目内容大多已暂存；所有既有内容视为用户工作，不做清理或重置。

## Agent A：代码审计结论

- `/tasks/detail` 的主文件是 `src/views/tasks/TaskDetail.vue`；当前结构仍是 `PageHeader + PageState + MetricGrid + 通用页签 + 单面板 + summary-rail`。
- P0 差距：默认不是生成结果；任务状态、标题、统计口径、11 个页签、九能力卡、待确认表、版本/导出预览、右侧追溯栏均缺失或错误。
- `/tasks/detail` 尚未进入 `AppLayout.vue` 的任务专属面包屑判断，且顶部仍会显示目标图没有的“演示数据”切换。
- 可复用：白色应用壳、九能力名称、Font Awesome、现有日志/抽屉基础方法和部分 mock；不复用通用 `DataTable`/通用 fixed drawer 强拼目标。
- 推荐边界：页面容器 + 任务头 + 摘要条 + 标签栏 + 结果卡网格 + 待确认表 + 版本导出预览 + 追溯栏；纯状态逻辑单独放入 `src/domain/taskDetail/taskResultState.js`。
- 关键风险：Store 的 localStorage 可能让旧数据覆盖新 mock；所有样式优先 scoped，避免改动被其他页面共用的 `.panel/.table-wrap/.drawer`。
- 局部验证：`node --test tests/taskDetailResultState.test.js`、任务详情结构契约测试、`npm run build` 和目标路由多视口浏览器检查。

## Agent C：交互与状态结论

- 初始状态固定为 9 类生成完成、15 条待确认、V1.0 草稿；第 9 类报告审核仍为待上传。
- 卡片、表格、顶部计数与抽屉必须由一份 `abilities + confirmationItems` 状态派生；确认/排除/说明操作必须幂等。
- 提交复核保持红色可点击，但有待确认项时点击后阻断并定位表格；归档任务才是真正 disabled。
- 单项确认/排除/编辑说明会同步更新待确认计数、行状态、能力卡状态、抽屉内容和操作日志。
- 从卡片“查看依据”时选中该能力首个待确认项；从表格进入时精确打开当前事项；抽屉五标签内容都必须随事项切换。
- 处理说明按目标图为选填；排除允许无说明，不能伪造默认原因。
- 15 条事项按 10 条/页分两页；表体可以内部滚动以匹配目标图首屏密度。
- 核心浏览器验收覆盖：重复操作幂等、批量处理边界、复核门禁、保存版本、局部导出、报告上传/检查、抽屉五标签、分页、Esc/键盘焦点。

## Agent B：视觉测量与响应式结论

- 全局骨架：左导航约 195px；顶栏 56px；中央外槽约 1059px；右追溯栏约 328px；中央与右栏间隔约 13px。
- 六个红框主区：任务摘要 1038×201；九能力结果 1029×381；待确认表 1027×208；版本时间线 459×139；输出预览 554×133；追溯栏 328×925。
- 摘要区三层高度约 83/75/41px；九卡采用 3×3、12px 横间距、10px 纵间距、统一约 108px 高；待确认表 5 行可见，每行约 22px。
- 目标风格：扁平白底、弱阴影、1px #E5EAF0、6px 卡片圆角；主链接 #1677FF；品牌红约 #B00000–#C00000；截图红框 #FA5151 不进入产品 UI。
- 响应式采用“三列 Grid + 密度变量 + 主区/追溯栏独立滚动”，不对中央区单独 `transform: scale()`；1366 宽仍保留 3 列卡，低于 1280 才折叠左栏并将右栏改为覆盖式。
- 1600×900/1440×900 参考密度 0.90，1366×768 参考密度 0.84；字体最小不低于 10px，关键正文不低于 11px。
- 避免空白：中央列必须 1fr，模块紧跟，不使用固定页面高度、空白占位或 `margin-top:auto`。

## 初步判断

- 目标是高密度桌面审计工作台；优先采用任务详情专属布局，避免把通用大卡片组件强行拼成目标图。
- 卡片、表格、右侧抽屉和顶部统计必须共享同一份“能力结果 + 待确认事项”状态，避免数量和状态漂移。
- 1366×768 不应把 3 列卡片和右抽屉随意改成单列；更可能需要基准画布同比缩放、适度压缩和受控纵向滚动的组合。

## 错误与风险记录

- 首次组合读取文件列表、Git 状态和图片时，过大的 Git/文件输出使工具调用失败；后续改用定向读取。
- 应用内浏览器 `Page.captureScreenshot` 持续超时；最终改用本机 Edge headless 捕获真实页面，再按实际 viewport 裁切，已生成同尺寸实现图与左右同图对照。
