# 任务详情“生成中”复刻设计

## 1. 目标状态

`/tasks/detail` 默认展示 `TASK-20250428001` 的资料解析完成后执行状态。用户打开页面即可看到任务进度、六步流程、九类能力、实时日志、当前模型步骤、来源快照、失败项、待处理项和执行明细，不需要在多个页签间寻找模型当前在做什么。

本轮基准是 1586×993 参考图。红框仅代表区域边界，不渲染到页面。初始视觉快照严格保留参考图的 `总进度 63%`、`完成 4 / 进行中 3 / 待执行 2 / 失败 0`，并同时保留九卡图面状态。由于参考图汇总与九卡字面计数不一致，汇总被定义为“源设计快照字段”，不从卡片数组重新计算；该取舍写入测试与注释，避免被误判为运行时统计错误。

## 2. 架构与文件边界

### `src/views/tasks/taskDetailExecutionData.js`

只负责目标页的确定性演示快照：任务元数据、页签、六步流程、四项汇总、九能力、实时日志、模型步骤、来源快照和待处理项。它不读取全局持久化 `db`，避免浏览器旧 localStorage 覆盖新页面。

### `src/views/tasks/TaskCapabilityExecutionGrid.vue`

负责左侧工作区：六步流程、四项汇总、3×3 能力卡和执行明细表。卡片与表格接收同一份 `capabilities`；组件不自行保存业务状态，只向父级发出 `view-log`、`view-result`、`run-background` 事件。

### `src/views/tasks/TaskExecutionLogRail.vue`

负责右侧三块：执行日志与阻断项、失败与重试、待处理提示。接收日志、当前能力步骤、来源快照和待办；向父级发出 `show-all-logs`、`view-source`、`handle-pending` 事件。

### `src/views/tasks/TaskDetail.vue`

负责任务头、元数据、总进度、版本、顶部动作、11 页签、双栏编排和轻量交互状态。初始路由无 `taskId` 时稳定使用目标任务；初始页签固定“分析过程”，但保留查询参数 `tab` 的兼容。

### `src/components/layout/AppLayout.vue`

仅增加 `/tasks/detail` 专属 shell class、面包屑“任务中心 / 任务详情”和详情页隐藏演示数据切换。其他任务路由、创建流程和解析页逻辑不变。

## 3. 视觉结构

### 应用壳

- 左导航约 207px；顶栏约 60px。
- 品牌红底白字，任务中心保持浅红激活背景与左侧红线。
- 主内容从约 x=220 开始；顶栏右侧保留通知、帮助和审计管理员。

### 任务头

- 面板约 1365×152px，分为标题动作层、元数据层、页签层。
- 标题 20–22px，旁边为橙色“生成中”。
- 元数据为任务编号、被审计单位、审计期间、任务类型、负责人、创建时间、总进度、当前版本。
- 总进度轨道约 127×6px；填充 63%。
- 顶部动作：后台运行、暂停任务、查看日志。
- 11 页签单行排列；“分析过程”红色并带 2px 下划线。

### 左主栏

- 宽约 1024px。
- 六步流程与四项汇总约 105px 高。
- 能力区约 387px 高，九卡严格三列三行；参考图第三列略宽，列比采用 `1fr 1fr 1.18fr`。
- 卡片约 115–117px 高，6px 圆角、1px 灰线、4px 进度条。
- 完成态绿色、进行中橙色、待执行灰蓝；当前阶段红色只用于任务主流程。
- 执行明细约 234px 高，采用专属紧凑表格，不复用全局 `DataTable`，避免全局最小宽度和行高破坏比例。

### 右栏

- 宽约 317px，与左侧流程顶对齐。
- 主日志面板约 472px，内部依次是 6 条实时日志、四步当前模型轨道、4 条来源快照。
- 失败重试约 61px，显示“暂无失败项”。
- 待处理约 205px，包含报告生成、报告审核和底部查看全部入口。

## 4. 响应式行为

- 1586×993：1:1 基准布局。
- 1440×900：页面壳按宽度约 0.908 缩放，并轻微收紧垂直间距。
- 1366×768：页面壳按宽度约 0.861 缩放，同时通过低高度令牌把任务头、流程、卡片和明细压缩约 10%；保留 3×3 和右栏同排，正文物理字号不低于约 10.5px，最多允许约 40px 纵向滚动。
- 1586px 以上停止整壳缩放。1920×1080 使用自然流，内容最大宽约 1600px，右栏最大 360px，卡片只增宽不增高。
- 1200px 以下进入功能性重排：右栏移到主区下方，能力卡两列；该档不属于像素级保真验收。
- 页面根永远不横向滚动；执行明细不足时仅表格容器局部滚动。

## 5. 交互设计

- “后台运行”：将对应能力标记为后台关注态，并显示不改变布局的 `aria-live` 提示；不启动定时器或伪造后端进度。
- “暂停任务”：切换本地暂停状态，按钮变为“继续任务”，状态标签和进度数值保持快照不跳变。
- 顶部“查看日志”：聚焦右侧日志区并清除能力过滤。
- 卡片“查看日志”：右栏只突出显示该能力相关日志；基准截图初始无过滤。
- “查看结果”：打开已有页面内抽屉，展示输出文件与来源摘要，不创建新路由。
- 待处理项：点击后打开同一抽屉，展示等待依赖和可执行动作；待执行按钮保持参考图禁用态。

## 6. 可访问性与质量约束

- 所有动作使用 `button` 或 `RouterLink`，图标按钮提供可访问名称。
- 当前步骤、状态和进度同时使用文字，不只靠颜色。
- 日志更新提示使用 `aria-live="polite"`，不自动抢焦点。
- 键盘可依次访问顶部动作、页签、卡片动作、来源链接和待处理项。
- 不新增依赖；图标使用现有 Font Awesome 与 `AuditIcon`。
- 不修改全局 `.panel`、`.btn`、`.table` 的基础规则；详情页全部使用专属 class。

## 7. 验证标准

- 数据测试证明目标快照完整且九能力顺序固定。
- 组件测试证明关键区域、11 页签、3×3 九卡、右栏和事件契约存在。
- 集成测试证明默认任务、初始页签、顶部动作和抽屉交互稳定。
- `npm run build` 成功。
- 浏览器在 1586×993、1366×768、1440×900、1600×1000、1920×1080 下无页面级横向溢出和控制台错误。
- 参考图与实现图同视口合并对照，修复全部 P0/P1/P2，`design-qa.md` 最终写明 `final result: passed`。
