# 调研、决策与迭代笔记

## 已确认事实

- 参考图为 1600×995，“已归档”只读状态。
- 当前 `/tasks/detail` 由 `PageHeader + PageState + MetricGrid + 通用页签 + 单内容面板 + 通用右栏` 组成，历史截图高度约 1615px。
- 当前 `TaskDetail.vue` 直接依赖通用 mock/store 记录，但不存在参考图所需的归档包专属数据组织。
- 当前 `AppLayout.vue` 没有为 `/tasks/detail` 提供参考图式的“任务中心 / 任务详情”紧凑面包屑和专属布局类。
- 项目是 Vue 3 + Vue Router + Vite，已包含 Font Awesome。
- app 仓库已有大量用户改动和未跟踪文件，本轮必须做手术式修改。
- 2026-07-10 实时浏览器基线（`1600×995`，`/tasks/detail?taskId=TASK-20250428001`）：`body/root` 实际为约 `1585×1578`，页面高度比目标多约 `583px`；首屏只有通用头部、办理状态、4 指标、页签和概览，参考图要求的成果卡、双表、复核/留痕、关联区和归档右栏均未呈现。
- 实时基线控制台无 error/warn；说明主要问题是结构和视觉，不是运行时异常。

## 并行 Agent 结论

- 文件实测 `1600×994`，与任务说明 995px 相差 1px；实现仍以 1600×995 为逻辑基准。
- 精确红框：标题 `216,69,1222,53`；元数据 `223,126,1100,60`；页签 `218,195,1103,44`；提示 `224,249,1094,40`；主框 `225,297,1094,690`；右栏 `1330,126,249,864`。
- 目标是高密度桌面后台：表格字号约 10–11px、行高 28–32px、弱阴影、4–6px 圆角；不能套用常见大卡片密度。
- 当前全局 `db` 会被 localStorage 恢复且不按 `taskId` 隔离；归档快照不应依赖它，也不应让查看动作污染不可变日志。
- 通用 `.section-grid/.table-wrap/table/.panel` 改动风险很高；本页必须路由类 + scoped 样式隔离。
- 归档页推荐单一快照驱动所有重复信息，确保 V1.2、归档时间、复核通过、6 份导出在各区一致。
- 响应式采用宽度缩放下限 0.85；1920 不放大画布，避免表格和卡片被拉松。
- 1366 使用局部滚动，不把右栏掉到下方，不把 10px 表格继续缩成不可读文字。

## 暂定判断

- 不能继续复用通用 `PageHeader/PageState/MetricGrid` 堆叠；它们的信息密度与目标完全不同。
- 归档页应使用专属数据对象驱动成果卡、双表、时间线和右栏，避免散落硬编码导致状态不一致。
- 基准桌面应维持主区约 1100px、右栏约 248px、两者间约 10px 的关系；宽屏限制最大画布宽度，窄屏同比缩放。

## 三种实现路线

1. **推荐：生命周期分派 + 归档专属页。** 新增归档快照和 `TaskDetailArchived.vue`，`TaskDetail.vue` 只负责分派；视觉、数据和交互高度隔离，其他状态可保留。
2. **覆盖式重写当前详情。** 文件最少，但会让草稿/生成中/待确认入口全部变成归档页，也会丢失旧状态能力，不采用。
3. **继续堆叠通用组件。** 初期改动小，但通用表格最小宽度、卡片密度和全局 CSS 会持续对抗目标，视觉难达标，不采用。

## 待用户确认的唯一口径

- 目标任务 `TASK-20250428001` 当前在任务列表仍标为“生成中”。推荐把它在任务列表也同步改为“已归档”，确保从列表进入详情时状态一致；这会使本轮额外触及 `taskCenterData.js` 和对应任务列表相关测试。

## 错误与风险记录

- 根仓库 `main` 无历史提交；app 子仓库有历史，但当前工作区已被大量修改，不能依赖破坏性回退。
- 参考图红框是标注层，不得被误实现为红色边框。
